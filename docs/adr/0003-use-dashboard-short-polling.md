# ADR-0003：教师 Dashboard 采用短轮询获取课堂状态

- 状态：已接受
- 日期：2026-07-23

## 背景

教师 Dashboard 需要持续显示参与者状态、活动时间和教师当前查看文件的最新已接受快照。学生插件主动向服务端提交数据，但浏览器 Dashboard 还需要一种方式获知服务端状态已经变化。

第一版面向同一局域网内不超过三十个参与者，只有教师 Dashboard 读取全班状态。正常连接时，从服务端接受代码快照到 Dashboard 显示该快照，目标约为两秒，可接受上限为三秒。

项目需要先学习普通 HTTP、Spring MVC 和浏览器请求模型。更新方式应容易观察、调试和恢复，且不应为了尚未验证的规模需求提前引入持久双向连接。

## 决策因素

- 在第一版课堂规模内满足三秒代码显示上限；
- 使用普通 HTTP 请求与响应，配合 Spring MVC 和原生 TypeScript 学习路线；
- Dashboard 断线后能够通过重新读取服务端权威状态恢复；
- 保持服务端和浏览器状态模型简单，减少连接生命周期处理；
- 允许通过课堂实验调整请求频率，而不把轮询间隔永久写死；
- 避免传输不必要的全班代码内容。

## 考虑过的方案

### 短轮询

Dashboard 按固定间隔向服务端发送普通 HTTP 请求，读取最新课堂状态。每次响应都是当时权威状态，因此断线恢复不需要补齐错过的事件序列。

短轮询实现和调试较直接，但会在没有变化时产生重复请求，显示延迟也受到轮询间隔、请求时间和渲染时间限制。

### Server-Sent Events

Server-Sent Events（SSE）允许服务端通过一个长连接向浏览器单向推送事件，方向符合服务端通知 Dashboard 的需求，也能减少无变化时的请求。

SSE 需要处理长连接、断线重连、事件标识以及重新同步权威状态。第一版尚未证明轮询无法满足延迟和负载目标，因此不先承担这些复杂度。

### WebSocket

WebSocket 提供持久双向通信，适合客户端和服务端都需要频繁主动发送消息的场景。CodePulse 的教师 Dashboard 主要读取状态，普通教师操作仍可使用独立 HTTP 请求；双向消息通道不是当前必要条件。

### 长轮询

长轮询让服务端暂时挂起请求，直到状态变化或请求超时后返回。它可以减少普通轮询的空响应，但会引入挂起请求、超时和立即重建请求等额外状态，同时仍然需要恢复性读取。

## 决策

CodePulse 第一版教师 Dashboard 使用短轮询，从教师服务端读取课堂概览和教师当前需要查看的数据。

具体轮询间隔、接口拆分、缓存和响应字段在实现阶段决定，并通过三十名参与者的课堂实验验证。所选参数必须使正常条件下的代码显示目标约为两秒，且不超过三秒；请求不得无界重叠或通过每次轮询下载所有参与者的全部代码。

学生插件仍在代码发生允许同步的文本修改后主动提交快照。该提交方式不属于本 ADR 的 Dashboard 轮询决定。

本决策不排除以后使用 SSE 或 WebSocket，但不会为了可能出现的未来需求同时维护多套实时更新机制。

## 影响

### 正面影响

- 第一版只需要理解和实现普通 HTTP 请求与响应；
- Dashboard 可以通过下一次成功请求恢复当前状态，不必补放错过的事件；
- Spring MVC 和浏览器原生请求 API 能够直接支持该模型；
- 轮询频率可以根据实际延迟、响应大小和教师电脑负载进行测量与调整。

### 负面影响

- 没有状态变化时仍会产生请求和响应；
- 更新不会在服务端接受后立即推送，最坏显示延迟受轮询间隔影响；
- 轮询过快会增加服务端、浏览器和局域网负载，过慢则无法满足显示延迟目标；
- 需要避免前一次请求尚未完成时不断堆积新的请求。

## 重新评估条件

出现以下情况之一时重新评估本决策：

- 在三十名参与者的课堂实验中无法同时满足三秒显示上限和可接受负载；
- 远程课堂或更高网络延迟使轮询产生明显带宽或响应问题；
- Dashboard 需要服务端主动发送不能等待下一次轮询的事件；
- 产品需要高频、双向或流式交互，且普通 HTTP 操作已不能清晰表达。

重新评估时优先比较 SSE 与 WebSocket 的实际需求匹配度，并通过新的 ADR 取代本文档，而不是仅因“实时”一词选择更复杂的协议。
