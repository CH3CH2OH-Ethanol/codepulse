# ADR-0004：代码同步采用完整文件快照

- 状态：已接受
- 日期：2026-07-23

## 背景

学生在课堂期间修改允许同步的文件时，VS Code 插件需要把最新代码状态发送给教师服务端。系统只需要让教师查看每个文件的最新已接受版本，不提供多人协作编辑、字符级过程回放或完整编辑历史。

第一版只同步所选工作区内实际发生文本修改且通过全部安全检查的文件，单文件上限为 `1 MiB`，课堂最多三十个参与者，并优先运行在局域网中。未保存到磁盘的编辑器文本也可能成为同步内容。

网络请求可能丢失、重复或乱序。同步表示方式需要允许后到的新状态修复此前丢失的更新，并与每个参与者、每个文件的单调递增版本配合。

## 决策因素

- 教师只需要可靠查看最新代码，而不是重放每次编辑；
- 丢失中间更新后，较新的数据应能够独立恢复当前状态；
- 客户端与服务端的版本、重试和错误处理应容易理解与测试；
- 不为多人协作编辑引入当前产品不需要的合并算法；
- 在局域网和当前文件大小边界内控制实现复杂度；
- 保留通过实际负载测试重新评估带宽成本的能力。

## 考虑过的方案

### 完整文件快照

插件在短暂防抖后发送文件当时的完整编辑器文本、工作区相对路径和快照版本。每个快照能够独立表示该版本的文件状态。

如果中间版本丢失，服务端仍能直接接受更高版本并得到正确的最新内容。相同版本可以幂等确认，较低版本则被拒绝，服务端不需要重建变化序列。

代价是文件中没有变化的部分也会重复传输。文件越大、修改越频繁，带宽、序列化和内存分配成本越高。

### 字符或行级差异

插件可以只发送相对于某个基础版本的插入、删除或替换操作。数据量可能小于完整快照，尤其适合大文件中的小修改。

但差异通常依赖明确的基础版本。一个中间差异丢失后，后续差异可能无法应用，需要补发、重新请求基线或执行冲突处理。客户端和服务端还必须对文本位置、换行与编码变化保持一致。

### 协作编辑算法

Operational Transformation 或 CRDT 等模型能够处理多个编辑者对同一文档的并发修改。CodePulse 不允许教师直接编辑学生代码，也不存在多个客户端共同写同一学生文件的需求，因此这些算法解决的是当前产品没有的问题。

### 仅在保存文件时同步

只监听磁盘保存可以降低发送频率，但学生可能长时间不保存，教师看到的代码会持续过期。这改变的是同步触发条件，而不是快照表示方式，并且不符合已确认的未保存文本同步行为。

## 决策

CodePulse 第一版使用完整文件快照同步代码。插件在允许同步的文本发生修改后执行短暂防抖，并发送该文件当时的完整编辑器文本、工作区相对路径和单调递增版本。

完整快照只表示一个实际修改的允许文件，不表示上传整个工作区。服务端只保留每个文件的最新已接受快照，不因该决定保存字符差异或完整编辑历史。

具体防抖时长、请求格式、文本编码处理和是否使用传输压缩在实现阶段决定。任何优化都不能改变工作区边界、文件拒绝规则、版本顺序或未保存文本告知要求。

## 影响

### 正面影响

- 每个快照能够独立恢复对应文件的最新状态；
- 丢失中间版本不会阻止更高版本成为当前状态；
- 重试、幂等和过期更新拒绝可以只依赖每文件版本；
- 客户端与服务端不需要维护差异基线、补丁序列或冲突合并；
- 测试可以直接比较服务端保存文本与客户端快照文本。

### 负面影响

- 每次更新都会重复传输文件中未改变的内容；
- 接近大小上限的文件被频繁修改时可能产生明显网络与处理负载；
- 防抖过短会增加重复传输，过长则可能无法满足 Dashboard 显示延迟目标；
- 未来远程课堂的网络条件可能使完整快照成本高于局域网环境。

## 重新评估条件

出现以下情况之一时重新评估本决策：

- 三十名参与者的课堂实验无法同时满足同步延迟和资源负载目标；
- 远程课堂的测量结果表明完整快照占用不可接受的带宽；
- 第一版文件大小边界发生显著扩大；
- 产品正式加入编辑历史、多人协作或其他需要操作序列的能力。

重新评估时应先测量实际文件大小、修改频率和压缩效果，再比较降低发送频率、传输压缩、分块或差异协议。不能仅因差异数据看起来更先进就承担额外状态与恢复复杂度。
