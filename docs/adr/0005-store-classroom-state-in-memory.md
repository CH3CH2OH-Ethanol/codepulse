# ADR-0005：第一版课堂状态只保存在教师服务端内存中

- 状态：已接受
- 日期：2026-07-23

## 背景

教师服务端需要保存课堂名称与配置、参与者状态、活动时间、每个文件的最新已接受快照和课堂结束流程所需的数据。第一版课堂是由教师创建并结束的临时会话，不提供跨课堂历史、作业归档或服务端重启恢复。

服务端运行在教师电脑上，不依赖租用的远程服务器。学生代码具有隐私敏感性；除教师明确选择导出最终 ZIP 外，应用不应主动在教师电脑上建立持久代码副本。

第一版已经接受教师服务端进程停止会导致当前课堂丢失。存储选择需要诚实呈现该限制，同时防止参与者通过大量文件或大体积快照无限占用服务端内存。

## 决策因素

- 保持课堂会话临时，并减少默认产生的学生代码副本；
- 避免在第一版引入数据库模式、迁移、清理和恢复流程；
- 支持不超过三十名参与者的最新状态查看和最终导出；
- 让服务端重启后的行为明确，而不是假装能够恢复不完整状态；
- 对参与者和整个课堂设置有限资源边界；
- 保留以后根据真实恢复需求加入本地持久化的可能。

## 考虑过的方案

### 进程内存

服务端使用进程内数据结构保存当前课堂权威状态。读取和更新直接，不需要数据库映射或磁盘事务；进程停止后状态消失，与第一版临时课堂边界一致。

该方案不能恢复崩溃或重启前的课堂，数据量也必须受到明确限制。内存中的数据还可能受操作系统交换空间、休眠和崩溃转储机制影响，因此只能承诺应用不主动持久化，不能承诺代码的物理字节绝不会接触磁盘。

### 嵌入式本地数据库

SQLite、H2 文件模式或类似嵌入式数据库可以在教师电脑上持久化课堂，使服务端重启后有机会恢复。代价是需要设计数据库模式、迁移、事务、损坏处理、过期清理和本地文件权限，并扩大隐私说明与测试范围。

### 普通文件持久化

服务端可以把课堂状态或每个快照直接写入 JSON 或普通文件。该方案初看比数据库简单，但仍需解决原子写入、并发更新、部分损坏、版本兼容、索引和清理，容易演变成缺少数据库保障的自制持久化系统。

### 远程数据库或托管服务

远程存储能够提供集中持久化和跨设备访问，但会引入服务器费用、互联网依赖、账户与密钥管理以及更大的学生代码隐私边界，不符合第一版本地、低成本和安装后离线运行的目标。

## 决策

CodePulse 第一版教师服务端只在当前进程内存中保存课堂代码、参与者状态和活动数据。应用除教师明确导出最终 ZIP 外，不主动把这些课堂内容持久化到数据库或普通文件。

教师服务端停止或崩溃后，系统不恢复原课堂，也不根据学生重连请求猜测已经丢失的状态。教师创建新课堂，学生重新确认授权并加入。

服务端必须为每名参与者的同步文件数量、每名参与者的最新快照总字节数和整个课堂的最新快照总字节数设置有限上限。具体默认值在实现前通过代表性项目和三十名参与者的负载实验确定；超过上限的快照被拒绝，不能覆盖此前已接受状态。

课堂结束后的最小终止标记也只保存在进程内存中，最长保留二十四小时或直到进程停止。插件为恢复身份而保存的版本和相对路径属于学生端本地状态，不改变本 ADR 的服务端存储决定。

## 影响

### 正面影响

- 第一版无需数据库、模式迁移和持久数据清理流程；
- 默认情况下，应用不会主动在教师电脑上长期保存学生代码；
- 当前课堂状态读取和替换逻辑较直接，适合学习领域状态与并发控制；
- 导出 ZIP 成为明确、可观察且由教师发起的持久化边界。

### 负面影响

- 服务端重启、崩溃或教师电脑断电会丢失整个进行中课堂；
- 未确认保存的最终导出也会随进程停止而丢失；
- 必须设计和测试内存容量上限、超限拒绝以及导出期间的峰值内存；
- 应用不主动持久化不等于操作系统保证内存内容永不进入交换空间或崩溃转储。

## 重新评估条件

出现以下情况之一时重新评估本决策：

- 真实课堂证明服务端重启恢复是必要能力，而不是偶发便利；
- 教师需要跨进程继续导出、查看或管理尚未清理的课堂；
- 内存容量实验表明最新快照状态无法在目标教师电脑上安全容纳；
- 产品进入远程、长期运行或多教师共享场景；
- 法律、学校政策或威胁模型要求不同的数据保留与删除保证。

重新评估时优先比较教师电脑上的受控本地持久化与嵌入式数据库，并单独设计加密、文件权限、迁移、保留期限和删除语义。加入持久化应通过新的 ADR 取代本文档，而不是悄悄改变隐私承诺。
